# Database migrations (Alembic)

Alembic is the **forward path** for schema changes. It targets the same database
as the app (URL comes from `Settings.async_database_url`, not `alembic.ini`) and
autogenerates against `app.db.Base.metadata`.

## Coexistence with the legacy in-app migrator

`app.db._migrate_schema` (the hand-maintained `ALTER TABLE … ADD COLUMN IF NOT
EXISTS` list) **still runs on startup** and remains the source of truth for the
current deployment. Alembic is introduced alongside it (baseline = the current
schema) so new changes can be authored as proper, reviewable, reversible
migrations. The two are not wired to run together automatically yet — cutting
over (having Alembic own `create_all`/migrations on boot and retiring
`_migrate_schema`) is a deliberate follow-up step to validate in staging, the
same conservative pattern used for the Redis store scaffolding.

## Common commands

Run from `backend/` (env vars must point at your DB, e.g. `DATABASE_URL=...`):

```bash
# Create a new migration from model changes
alembic revision --autogenerate -m "add foo to bar"

# Apply migrations
alembic upgrade head

# Roll back one
alembic downgrade -1

# Show current / history
alembic current
alembic history
```

## Authoring notes

- The env runs with `render_as_batch=True` on SQLite so `ALTER` operations work
  there (SQLite has limited native ALTER support).
- `compare_type=True` is enabled, so column-type changes are detected.
- Always review autogenerated migrations — Alembic can miss server defaults,
  custom types, and some index/constraint nuances.
